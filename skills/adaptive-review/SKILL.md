# 自适应评审 (adaptive-review v2.1)

## 概述

编排自适应多智能体评审团队，根据任务特征、风险因子和复杂度选择最优协作模式与执行路径，通过结构化工作流（事实收集 - 交叉辩论 - 共识收敛）产出可操作的洞察和行动计划。

v2.1 在 v2 基础上的核心改进：
- **强制审批门禁**：Phase 0 后必须等待用户确认，除非用户明确豁免
- **风险升级因子**：认证、支付、数据删除等高风险变更自动升级评审深度
- **自适应镜头**：轻量模式不再使用固定角色，根据变更内容动态选择评审视角
- **门禁恢复工作流**：每轮质量门禁失败时有明确的恢复路径，而非简单重做
- **证据分级框架**：E1-E4 四级证据分类，P0/P1 发现必须依赖 E1/E2 级证据
- **误报抑制**：每个 P0/P1 发现必须回答触发条件、频率、已有防护
- **严重度校准**：P0-P3 明确定义，必须附带影响声明和置信度
- **评审经济学层**：行动项增加杠杆率（风险降低 / 工程成本），按杠杆排序
- **LEARNED_RULES 结构化**：每条规则必须包含 source/confidence/expires_at 等元数据
- **Local-Global 对齐强化**：所有评审者提交前必须完成对齐检查

## 适用场景

- 项目设计评审、架构评估
- 产品策略分析（需要多方博弈视角）
- 代码质量审计（需要跨职能交叉验证）
- 用户明确要求多角度或团队协作式分析的软件项目评审任务

## 不适用场景

- 单一维度的快速代码审查（直接用 architect-reviewer agent 更高效）
- 非软件项目的分析任务（本 skill 的角色和 prompt 面向软件项目设计）
- 需要实际执行代码修改的任务（本 skill 只产出分析报告和行动计划）
- 简单问题的快速问答（多 agent 编排的启动成本远大于收益）

## 三种模式的实质差异

| 维度 | 集中调度 | 领域主导 | 对等协作 |
|------|---------|---------|---------|
| Round 1 agent 数 | 3（并行） | 3（Lead 加深，其余精简） | 3（并行，对等） |
| Round 2 agent 数 | **1**（仅评审组） | **2**（Lead 交叉 + 评审组） | **4**（3 交叉 + 评审组） |
| Round 2 交叉评审 | Team Lead 自行完成 | Lead 评审全部，其余不参与 | 三方互评 |
| Round 3 决策方式 | Team Lead 直接裁决 | Lead 综合裁决 | 投票矩阵 + 共识收敛 |
| 适用场景 | 时间紧、产品决策、不确定选哪个模式 | 某领域需要特别深入 | 跨职能复杂权衡 |

## 任务复杂度分级（含风险升级）

Phase 0 同时评估：协作模式、执行深度、风险等级。

### 基础复杂度

| 复杂度 | 判定标准 | 执行路径 | Agent 总数（集中调度） |
|--------|---------|---------|---------------------|
| 轻量 | < 10 文件 或 单一模块 | Round 1（自适应镜头）+ Team Lead 直接裁决 | 2-3 |
| 标准 | 10-50 文件 或 2-5 模块 | 完整 Round 1-3 | 4 |
| 深度 | > 50 文件 或 > 5 模块 或 跨服务 | 完整 Round 1-3 + 可选 Round 2.5 | 4-5 |

### 风险升级因子

**风险因子清单：** 认证、授权、租户隔离、支付、计费、工作流引擎、审批流、状态机、数据库迁移、Schema 变更、公共 API 契约、生产事故修复、安全边界、数据删除、批量更新。

**升级规则：**
- 出现 **1 个**风险因子：轻量 → 标准
- 出现 **2 个或以上**风险因子：直接 → 深度
- **风险升级优先于文件数量判断**

### 自适应镜头（轻量模式专用）

轻量模式不使用固定 PM/Designer/Engineer 角色，改为动态镜头选择：

**核心评审者**：始终存在，负责整体事实收集。

**风险镜头**（可选 1 个）：
- Security（安全）
- Performance（性能）
- Reliability（可靠性）
- Data Correctness（数据正确性）
- UX Regression（UX 回归）
- API Compatibility（API 兼容性）

**上下文镜头**（可选 1 个）：
- Frontend（前端）
- Backend（后端）
- Database（数据库）
- DevOps（运维）
- Domain Expert（领域专家）

Team Lead 根据变更内容自动选择镜头，输出选择理由。
示例："Frontend UI 变更 → UX Regression 镜头 + Frontend 上下文镜头"

### 条件性轮次跳过规则

- Round 1 事实收集显示**无跨领域争议**（三个视角无交集） → Round 2 降级为评审组-only
- Round 2 争议密度 < 20%（分歧项/总议题数） → Round 3 直接裁决，跳过投票矩阵
- Round 2 争议密度 > 50% → 考虑增加 Round 2.5（对争议领域补充调研）

## 工作流

### Phase 0：模式选择 + 复杂度评估 + 风险分析

**步骤 1 - 评估任务复杂度：**
分析目标范围（文件数、模块数、变更行数），确定基础执行深度。

**步骤 2 - 扫描风险升级因子：**
检查变更是否涉及风险因子清单中的任何领域。按升级规则调整深度。

**步骤 3 - 选择协作模式：**
分析任务特征，选择协作模式。详见 `references/mode-selection.md`。

- **集中调度模式**（默认）
- **领域主导模式**
- **对等协作模式**

**步骤 4 - 选择自适应镜头（仅轻量模式）：**
如果是轻量模式，Team Lead 根据变更内容选择风险镜头和上下文镜头。

**【强制审批门禁】（Enhancement 1）：**
Phase 0 完成后，**必须停下来**向用户展示审批面板：

```
=== 评审审批面板 ===
复杂度级别：轻量 / 标准 / 深度（含风险升级因子，如有）
协作模式：集中调度 / 领域主导 / 对等协作
选定 Agent：列出角色和数量
预估 Token：XXX
执行轮次：Round 1 / Round 1-2 / Round 1-3
跳过轮次：Round 2（理由：xxx）/ 无
自适应镜头：[仅轻量模式] 风险镜头=xxx, 上下文镜头=xxx
====================
```

**然后停止，等待用户批准。**

**例外：** 如果用户明确说了以下之一，可跳过审批直接执行：
- "continue automatically"
- "run full review"
- "no confirmation needed"

### Phase 1：Round 1 - 事实收集（并行）

启动 Explore agent 并行执行。角色 prompt 从 `references/role-prompts-r1.md` 加载。

**标准/深度模式：** 启动 PM / Designer / Engineer 三个 agent。
**轻量模式：** 启动核心评审者 + 选定的镜头 agent（2-3 个 agent）。

**关键约束：** 只写事实，不做评价，不给建议。禁用词："好"、"差"、"应该"、"建议"、"改进"、"更好"。

**证据分级（Enhancement 5）：** 每条事实必须标注证据级别：
- **E1（直接证据）**：代码、diff、测试、日志、运行时输出
- **E2（间接证据）**：文档、配置、契约、设计文档
- **E3（结构推理）**：架构推理、依赖推理
- **E4（专家假设）**：经验性判断

**完成后 Team Lead 动作：**
1. 收集所有事实清单
2. 运行 **Round 1 质量门禁**：
   - [ ] 每条事实是否包含来源（文件路径:行号）？
   - [ ] 每条事实是否标注证据级别（E1-E4）？
   - [ ] 是否零评价性词汇？
   - [ ] 是否覆盖了 prompt 要求的所有调研维度？
3. **门禁恢复工作流（Enhancement 4）：** 不通过时按以下路径恢复：
   - 证据缺失 → 返回原评审者补充
   - 维度缺失 → 启动针对性评审者补测
   - 严重度不明 → 请求校准
4. 复制画布模板到项目的 `reviews/` 目录
5. 提炼关键事实写入画布 Round 1 区域
6. 评估是否需要完整 Round 2（条件性轮次跳过规则）

### Phase 2：Round 2 - 交叉辩论（按模式执行）

所有 agent 读取共享画布。角色 prompt 从 `references/role-prompts-r2.md` 加载。

**增强评审组（挑战者 A + 挑战者 B + 审计者）：**

**误报抑制（Enhancement 6）：** 评审组对每个 P0/P1 发现必须验证：
1. 用户是否真的能遇到这个问题？
2. 触发条件是什么？
3. 发生频率如何？
4. 是否已有防护措施？
5. 修复成本是否低于风险成本？

无法回答 → 降级置信度，可能降低严重度。不推广推测性发现。

**完成后 Team Lead 动作：**
1. 收集所有报告
2. 运行 **Round 2 质量门禁**：
   - [ ] 每个反驳是否附带理由？
   - [ ] 评审组是否完成误报抑制检查？
   - [ ] 关键质询是否标记（0-3 条）？
   - [ ] 争议密度是否已计算？
3. **门禁恢复工作流：**
   - 挑战未解决 → 触发 Round 2.5
   - 持续分歧 → 标记为 Open Risk（不强制共识）
4. 构建投票矩阵，写入画布 Round 2 区域

### Phase 3：Round 3 - 共识收敛（Team Lead 执行）

由 Team Lead 直接执行，不委派给子 agent。

**严重度校准（Enhancement 7）：**

| 级别 | 定义 | 要求 |
|------|------|------|
| **P0** | 灾难性：安全漏洞、数据损坏、完全宕机 | 必须附带影响声明 + E1/E2 证据 |
| **P1** | 重大功能故障：严重业务影响 | 必须附带影响声明 + E1/E2 证据 |
| **P2** | 中等风险：降级行为 | 必须附修理由 |
| **P3** | 可维护性：优化、代码质量 | 仅记录，不生成 Issue |

**评审经济学层（Enhancement 12）：** 每个行动项必须计算：

```
工程成本：Low / Medium / High
风险降低：Low / Medium / High
杠杆率 = 风险降低 / 工程成本
```

Team Lead 按杠杆率排序行动项，优先高杠杆（低成本高收益），避免低杠杆（高成本低收益）。

**"继续迭代价值"判断：**
- 如果 Round 2 未产生实质争议 → 直接输出共识 + 行动计划
- 如果存在高争议议题 → 按模式执行完整收敛流程
- **未解决的风险保持可见** — 不强制共识，允许 Open Risk 存在

**完成后运行 Round 3 质量门禁：**
- [ ] 每个行动项包含成本估算 + 杠杆率？
- [ ] 每个行动项标注负责角色？
- [ ] P0/P1 发现基于 E1/E2 证据？
- [ ] 关键质询逐条回应？
- [ ] 未解决分歧标记为 Open Risk？

### Phase 4：交付

1. 将完整 Round 3 结果写入共享画布
2. 根据用户偏好输出格式：
   - **画布格式**（默认）：完整 markdown 画布文件
   - **GitHub Issue 格式**：仅 P0/P1 生成 Issue（P2 按需，P3 不生成）
   - **ADR 格式**：仅在存在架构决策 + 评估了备选方案 + 有长期后果时生成
3. 向用户呈现简明摘要

## 证据分级框架

所有发现必须标注证据级别（Enhancement 5）：

| 级别 | 类型 | 可信度 | 适用 |
|------|------|--------|------|
| **E1** | 直接证据（代码/diff/测试/日志） | 最高 | P0/P1 发现必须主要基于此 |
| **E2** | 间接证据（文档/配置/契约） | 高 | 可支撑 P0/P1 |
| **E3** | 结构推理（架构/依赖推理） | 中 | 必须显式标注 |
| **E4** | 专家假设（经验判断） | 低 | 必须显式标注，不可单独支撑 P0 |

**发现格式：**
```
发现标题：
证据级别：E1/E2/E3/E4
证据来源：文件路径:行号
置信度：High / Medium / Low
```

## LEARNED_RULES 规范（Enhancement 10）

每条学习规则必须包含完整元数据，无元数据的规则不被注入：

```
规则内容：xxx
来源：上次评审 / 历史模式
置信度：High / Medium / Low
创建时间：YYYY-MM-DD
过期时间：YYYY-MM-DD（防止陈旧偏见）
适用上下文：前端变更 / 后端变更 / 安全审查 / ...
```

**Team Lead 职责：** 注入前必须评估每条规则与当前评审的相关性。过期的、不相关的规则不注入。

## Local-Global 对齐检查（Enhancement 11）

所有评审者提交输出前必须完成：

```
【对齐检查】
1. 另一个角色是否能合理地以不同方式解读你的输出？
2. 如果是，你已澄清了哪些假设？
3. 你已标明了哪些不确定之处？
```

不通过对齐检查的输出不进入下一轮。

## 核心设计原则

1. **风险优先于规模** — 高风险变更即使文件少也升级评审深度
2. **复杂度决定深度** — 小型评审不启动完整流水线
3. **事实先于观点** — Round 1 纯事实收集，附带证据级别
4. **质量门禁不可跳过** — 每轮检查清单是强制步骤，有恢复路径
5. **证据分级** — P0/P1 必须基于强证据，弱证据显式标注
6. **误报抑制** — 每个发现必须回答"用户真能遇到吗"
7. **不强制共识** — 未解决风险保持为 Open Risk，可见但不纳入行动计划
8. **杠杆优先** — 按风险降低/工程成本排序，避免低收益建议
9. **规则有保质期** — LEARNED_RULES 必须有 expires_at，防止陈旧偏见
10. **输出可操作** — Issue 仅 P0/P1，ADR 仅架构决策

## 常见错误

| 错误 | 解决 |
|------|------|
| 跳过 Phase 0 审批门禁 | 必须展示审批面板并等待确认（除非用户明确豁免） |
| 忽略风险升级因子 | 风险因子优先于文件数量判断 |
| P0 发现仅基于 E4 证据 | P0/P1 必须主要基于 E1/E2，E3/E4 必须显式标注 |
| 未运行误报抑制检查 | 每个发现必须回答 5 个验证问题 |
| 门禁失败后强制重做 | 按恢复路径处理：证据缺失→补充，维度缺失→针对性补测，挑战未解决→Round 2.5 |
| 强制达成共识 | 未解决分歧标记为 Open Risk，保持可见 |
| LEARNED_RULES 无元数据 | 无元数据的规则不注入 |
| 为 bug fix 生成 ADR | ADR 仅用于架构决策 |
| P3 发现生成 Issue | Issue 仅生成 P0/P1，P2 按需，P3 不生成 |
