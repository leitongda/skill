> v2.1 变更：增加门禁恢复工作流、证据分级、误报抑制、严重度校准、评审经济学层。

## 质量门禁总览（含恢复路径）

每轮完成后必须运行对应的检查清单。**不通过则按恢复路径处理，不强制重做。**

### Round 1 质量门禁

- [ ] 每条事实是否包含来源（文件路径:行号）？
- [ ] 每条事实是否标注证据级别（E1-E4）？
- [ ] 是否零评价性词汇？（自动扫描禁用词）
- [ ] 是否覆盖了 prompt 要求的所有调研维度？
- [ ] 三个视角的事实是否有交集？（有交集 → 需要交叉辩论）

**恢复路径（Enhancement 4）：**

| 失败原因 | 恢复动作 |
|---------|---------|
| 证据缺失（来源不完整） | 返回原评审者补充具体来源 |
| 维度缺失（调研不完整） | 启动针对性评审者仅补测缺失维度 |
| 严重度不明 | 请求 Team Lead 校准 |
| 评价性词汇混入 | 要求去除评价、保留事实部分 |

### Round 2 质量门禁

- [ ] 每个反驳是否附带理由？
- [ ] 评审组是否完成误报抑制检查（5 问验证）？
- [ ] 评审组是否标记了关键质询（0-3 条）？
- [ ] 交叉评论是否覆盖了其他两方的主要发现？
- [ ] 争议密度是否已计算？

**恢复路径（Enhancement 4）：**

| 失败原因 | 恢复动作 |
|---------|---------|
| 反驳无理由 | 要求补充理由 |
| 未完成误报检查 | 评审组补完 5 问验证 |
| 挑战未解决 | 触发 Round 2.5（补充调研） |
| 持续分歧 | 标记为 **Open Risk**（不强制共识） |

### Round 3 质量门禁

- [ ] 每个行动项是否包含成本估算 + 杠杆率？
- [ ] 每个行动项是否标注负责角色？
- [ ] P0/P1 发现是否基于 E1/E2 证据？
- [ ] 关键质询是否逐条回应？
- [ ] 回应不充分的关键质询是否归入"待定"？
- [ ] 未解决分歧是否标记为 Open Risk？

**恢复路径：**

| 失败原因 | 恢复动作 |
|---------|---------|
| 行动项缺成本/杠杆率 | Engineer 补充估算 |
| P0/P1 仅基于 E3/E4 | 降级为 P2 或补充 E1/E2 证据 |
| 关键质询未回应 | 标记为"待定：需用户确认" |

## 条件性轮次跳过

### 跳过 Round 2 的条件（轻量级执行）

以下**全部**满足时可跳过 Round 2：
1. 任务复杂度为"轻量"
2. Round 1 三个视角的事实**无交集**（无跨领域争议）
3. 未触发风险升级因子
4. Team Lead 判断无需交叉验证

跳过后直接进入 Round 3 Team Lead 裁决。

### Round 2 内降级的条件

- Round 1 事实交集极少 → Round 2 仅启动评审组，跳过交叉评论
- 对等协作模式下，如三方事实高度一致 → 交叉评论可合并为 1 个 agent

### Round 3 简化的条件

- Round 2 争议密度 < 20% → 直接裁决，跳过投票矩阵
- Round 2 无关键质询 → 共识项直接纳入行动计划

## 证据分级框架（Enhancement 5）

所有发现必须标注证据级别：

| 级别 | 类型 | 可信度 | 约束 |
|------|------|--------|------|
| **E1** | 直接证据（代码/diff/测试/日志/运行时输出） | 最高 | P0/P1 必须主要基于此 |
| **E2** | 间接证据（文档/配置/契约/设计文档） | 高 | 可支撑 P0/P1 |
| **E3** | 结构推理（架构推理/依赖推理） | 中 | 必须显式标注为 E3 |
| **E4** | 专家假设（经验性判断） | 低 | 必须显式标注为 E4，不可单独支撑 P0 |

**发现格式：**
```
发现标题：XXX
证据级别：E1 / E2 / E3 / E4
证据来源：文件路径:行号
置信度：High / Medium / Low
```

**规则：**
- P0 和 P1 发现应主要基于 E1 或 E2 证据
- E3 和 E4 发现必须显式标注
- 仅基于 E4 的发现不可标记为 P0

## 误报抑制机制（Enhancement 6）

评审组对每个 P0/P1 发现必须完成 5 问验证：

1. **用户能否实际遇到？** — 用户是否真的能触发这个问题？
2. **触发条件是什么？** — 具体的触发路径
3. **发生频率如何？** — 低频/中频/高频
4. **是否已有防护？** — 现有的缓解措施
5. **修复成本 vs 风险成本？** — 修复是否值得

**处置：**
- 5 问均可回答 → 保持原严重度
- 1-2 问无法回答 → 降级置信度（High → Medium）
- 3+ 问无法回答 → 降低严重度（P0 → P1, P1 → P2）
- 完全推测性发现 → 不推广，标记为"需验证"

## 严重度校准标准（Enhancement 7）

| 级别 | 定义 | 必要条件 |
|------|------|---------|
| **P0** | 灾难性：安全漏洞、数据损坏、完全宕机 | 影响声明 + E1/E2 证据 + 置信度评分 |
| **P1** | 重大功能故障：严重业务影响 | 影响声明 + E1/E2 证据 + 置信度评分 |
| **P2** | 中等风险：降级行为 | 修理由 + 置信度评分 |
| **P3** | 可维护性：优化、代码质量 | 仅记录，不生成 Issue |

**P0/P1 必须附带：**
- 严重度级别 + 定义匹配理由
- 影响声明（谁受影响、影响什么）
- 置信度评分（High / Medium / Low）

## 评审经济学层（Enhancement 12）

每个行动项必须计算杠杆率：

```
工程成本：Low（< 4h）/ Medium（4-16h）/ High（> 16h）
风险降低：Low / Medium / High
杠杆率 = 风险降低 / 工程成本
```

**杠杆率映射：**

| 工程成本 | 风险降低 | 杠杆率 | 优先级建议 |
|---------|---------|--------|-----------|
| Low | High | **最高** | P0 |
| Low | Medium | 高 | P0-P1 |
| Medium | High | 高 | P1 |
| Medium | Medium | 中 | P1-P2 |
| High | High | 中 | P2 |
| High | Medium | 低 | P3 或不推荐 |
| Low | Low | 低 | P3 或不推荐 |

**Team Lead 排序规则：** 按杠杆率降序排列行动项。避免高成本低收益的建议。

## 增强评审组（替代单一 Critic）

### 角色定义

| 角色 | 职责 | 特殊职责 |
|------|------|---------|
| **挑战者 A** | 反对角度，寻找被高估的优势 | 执行误报抑制检查 |
| **挑战者 B** | 替代方案角度，"换一种做法" | 检查证据级别是否支撑严重度 |
| **审计者** | 逻辑一致性、数据充分性 | 验证发现格式的证据完整性 |

### 按模式的评审组配置

| 模式 | 实际 Agent 数 | 说明 |
|------|-------------|------|
| 集中调度 | 1 | 三个角色合并为 1 个 agent 的 prompt |
| 领域主导 | 1-2 | 挑战者+审计者合并为 1 个，或拆分为 2 |
| 对等协作 | 2-3 | 挑战者 x2 + 审计者，可独立运行 |

### 关键质询标记规则

- 每个挑战者最多标记 1 条关键质询
- 审计者最多标记 1 条关键质询
- 总计不超过 3 条
- 标记标准：涉及根本性逻辑缺陷或重大遗漏

|------|---------|---------|---------|---------------|-------------|

## Round 1 协议：事实收集

### 通用规则

1. 三个 Explore agent 并行启动（PM、Designer、Engineer），或轻量模式下使用自适应镜头
2. 每个 agent 只负责自己的视角领域
3. 输出必须是结构化事实清单，每条附带来源（文件路径:行号）和证据级别（E1-E4）
4. 禁止使用评价性词汇
5. 所有 agent 使用 run_in_background: true 并行执行
6. Prompt 中的 LEARNED_RULES 由 Team Lead 评估相关性后注入（仅注入含完整元数据的规则）
7. 提交前完成 Local-Global 对齐检查

### 完成标志

三个 agent 全部返回事实清单 + Round 1 质量门禁通过（含恢复路径完成）。

---

## Round 2 协议：交叉辩论

### 目标

交叉验证事实，发现跨领域问题，挑战薄弱论点，抑制误报。

### 通用规则

1. 所有 agent 读取共享画布
2. 对每个发现验证证据级别是否支撑其严重度
3. 对每个 P0/P1 发现执行误报抑制 5 问验证
4. 完成标志：所有 agent 返回 + Round 2 质量门禁通过

### 不强制共识原则

- 持续分歧标记为 **Open Risk**，保持可见
- Open Risk 不纳入行动计划，但记录在画布中
- 允许评审以"部分共识 + Open Risks"结束

---

## Round 3 协议：共识收敛

### 执行者

Team Lead（非 agent），直接在主会话中执行。

### 行动计划格式（含经济学层）

按杠杆率排序：
- **P0**：灾难性，立即修复（杠杆率最高）
- **P1**：重大故障，本轮修复
- **P2**：中等风险，下个周期规划
- **P3**：可维护性，仅记录
- **Open Risk**：未解决分歧，保持可见
- **待定**：需用户确认

每个行动项必须包含：
- 具体行动描述
- 严重度级别 + 影响声明（P0/P1 必需）
- 证据级别（E1-E4）+ 证据来源 + 置信度
- 工程成本 + 风险降低 + 杠杆率
- 负责角色

### 输出格式选项

| 格式 | 适用场景 | 生成规则 |
|------|---------|---------|
| 画布（默认） | 所有场景 | 完整 markdown 画布文件 |
| GitHub Issue | P0/P1 默认，P2 按需 | 仅 P0/P1 自动生成，P2 用户请求时生成，P3 不生成 |
| ADR | 架构决策 | 仅当：存在架构决策 + 评估了备选方案 + 有长期后果时生成 |

### GitHub Issue 生成规则（Enhancement 8）

仅生成 P0/P1 Issue。P2 仅在用户请求时生成。P3 不生成 Issue。

Issue 模板字段：
- 标题
- 严重度（P0/P1/P2）
- 标签
- 负责人
- 风险描述
- 验收标准
- 证据（来源 + 级别）
- 预估工作量

### ADR 生成规则（Enhancement 9）

仅在满足以下**全部**条件时生成 ADR：
1. 存在架构决策（非 bug 修复）
2. 评估了备选方案
3. 存在长期后果

ADR 必须包含：
- 背景
- 决策
- 评估的备选方案
- 后果
- 评审共识
